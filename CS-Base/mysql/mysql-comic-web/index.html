05 / Buffer Pool

Buffer Pool 是 MySQL 的热数据客厅

磁盘太远，InnoDB 会把常用的数据页、索引页搬进内存。改数据时，先改内存页，标成脏页，之后再刷盘。

SQL 需要数据页
Buffer Pool 有吗？
有 → 直接读内存 ⚡
没有 → 从磁盘读页 → 放入 Buffer Pool
被修改 → 脏页 → 稍后刷盘

06 / Logs

UPDATE 背后的三本账

1写 undo log：留下旧值，方便回滚和 MVCC
2改 Buffer Pool：先改内存里的页
3redo prepare：InnoDB 准备提交
4写 binlog：Server 层记录归档和复制日志
5redo commit：事务真正闭环

undo log

像橡皮擦：回滚、读旧版本。

redo log

像保险箱：崩溃恢复，不丢已提交事务。

binlog

像录像带：主从复制、归档恢复。

08 / Locks

锁：别让两个人同时改同一张卡片

普通快照读？
通常走 MVCC，不加行锁
当前读 / 修改？
需要行级锁
唯一索引等值命中？
Record Lock 锁具体记录
范围 / 非唯一索引？
Next-Key Lock = 记录锁 + 间隙锁

关键分析

  • InnoDB 的行锁本质上加在索引上。
  • 查询条件没走索引，扫描更多记录，锁范围可能变大。
  • 死锁不是数据库坏了，而是事务互相等锁；短事务、固定访问顺序、失败重试是常见解法。

09 / Debug

慢 SQL 排查地图

点一个排查项，看看提示。

最后 6 句话

  1. SQL 先经过 Server 层，再交给 InnoDB。
  2. InnoDB 以页为单位读写，Buffer Pool 用来缓存页。
  3. B+ 树优势是树高低、少 I/O、范围查询强。
  4. MVCC 靠 Read View 和 undo log 版本链。
  5. redo log 防崩溃，binlog 做复制和归档。
  6. 锁要结合索引理解：索引走得好，锁通常更精准。
